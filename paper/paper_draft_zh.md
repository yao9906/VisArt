# VisArt：基于可视化设计知识检索增强的大模型图表生成与自动化质量评估系统

---

## 摘要

大型语言模型（LLM）已展现出自动生成可视化代码的能力，但其生成结果的设计质量难以保障。现有工作存在两个尚未解决的问题：一是设计知识的组织和检索方式不够充分，LLM 难以系统性地获取可视化领域积累的设计准则；二是缺乏针对 LLM 生成结果的自动化质量检测机制，尤其是面向 D3.js 等代码驱动的可视化。本文提出 VisArt，从知识检索增强和生成后质量评估两个方面提供系统性解决方案。在知识检索方面，我们构建了以知识图谱组织的可视化设计知识库，通过图遍历发现与用户查询直接和间接相关的设计知识，并经约束推理引擎生成结构化上下文注入 LLM 提示词。在质量评估方面，我们提出 Design Lint——一个面向 D3.js 可视化的三层自动化质量检测系统，依次从代码层面、运行时渲染层面和设计规则层面进行审查，输出量化的设计质量分数（DQS），并在分数低于阈值时自动触发修复。基于两个公开数据集的案例分析表明，VisArt 能够有效提升 LLM 生成可视化的设计质量，并为设计决策提供透明、可追溯的依据。

---

## 第一章 引言

大型语言模型在代码生成方面的进展，已将其能力拓展到可视化图表代码的自动合成领域。用户仅需以自然语言描述分析需求，LLM 即可在数秒内生成可运行的可视化代码，从而将传统上高度依赖编程技能的图表创建过程大幅简化。围绕这一能力，研究界已积累了丰富的成果。LIDA [1] 将可视化生成拆解为目标理解、规范确定与代码合成三个阶段。LightVA [2] 率先采用 Agent 架构，让 LLM 承担可视分析的任务规划角色，自动协调数据处理、图表渲染与洞察提取。ProactiveVA [3] 进一步赋予系统主动感知分析进程的能力。SmartMLVs [4] 将生成范围拓展到多个协调联动的交互式视图。

然而，上述系统在设计质量保障方面存在两个共性不足，构成了本文的研究动机。

**其一，设计知识的组织与检索方式有待改进。** 可视化研究已积累了数十年的设计准则——从 Cleveland 和 McGill [7] 的视觉通道感知精度排名，到 Borland 和 Taylor [8] 关于彩虹色图危害的实验证据，再到 WCAG 2.1 [9] 的无障碍对比度标准。然而，LLM 获取这些知识的途径仍然有限：或是依赖训练过程中习得的参数化知识（可靠性不足），或是通过 RAG 从外部知识库逐条检索（难以捕获设计知识各维度之间的关联性）。具体而言，可视化设计知识具有显著的关联性——选择某种图表类型通常连带涉及编码策略、颜色方案和交互设计等多个维度的考量，而现有的逐条文本检索难以表达这些维度之间的结构化关联。

**其二，缺乏针对 LLM 生成结果的自动化质量检测。** 即便向 LLM 注入了设计规则，模型仍可能忽视或偏差应用这些规则。然而，当前没有自动化机制在图表生成后对其进行质量检查和评分。现有的可视化质量检测工具（如 VizLinter [10]、GeoLinter [11]）均面向 Vega-Lite 等声明式规范设计，可以直接从 JSON 配置中读取编码映射来判断对错。但 LLM 越来越多地生成 D3.js 这类代码驱动的可视化，其视觉效果取决于 JavaScript 的运行时执行，无法像解析 JSON 配置那样直接推断设计意图——例如，同一段散点图代码在 100 条数据下运行正常，在 3,000 条数据下可能出现严重的标记重叠，而这种问题只有在代码实际运行并渲染出 SVG 后才可量化。

本文提出 **VisArt**，围绕上述两个问题提供系统性的解决方案。

**贡献一：基于图结构知识库的可视化设计知识检索增强。** 我们构建了一套涵盖编码规则、颜色感知、交互策略和任务分类等维度的可视化设计知识库，并以知识图谱的形式组织各类知识之间的关联关系。检索时，系统不仅找到与查询直接相关的设计知识，还沿着图谱的关联边发现间接相关的技术与规则。检索结果进一步经约束推理引擎处理，生成针对当前数据特征和分析任务的结构化上下文注入 LLM 提示词。

**贡献二：面向 D3.js 的三层自动化图表质量评估与纠错。** 我们提出 Design Lint 系统，依次从代码层面、渲染结果层面和设计规则层面对 LLM 生成的可视化进行自动审查，综合输出设计质量分数（DQS, 0-100）。当分数低于阈值时，系统将检测结果反馈给大模型进行定向修复，形成"生成→检测→修复"的自动化闭环。

此外，我们贡献了评估基准 **VisArt-Bench**，包含 12 项测试任务与 6 项量化指标，用于对 LLM 可视化生成系统进行可复现的对比评估。

**关于 D3.js 的选择。** 与多数现有系统生成 Matplotlib 静态图表或 Vega-Lite 声明式规范不同，VisArt 以 D3.js 为目标代码。这一选择基于三方面考量。第一，D3 的表达力远强于声明式语法，能实现力导向图、弦图、语义缩放等复杂可视化形式 [12]，而 LLM 的代码生成能力弥补了 D3 在编码复杂度方面的传统痛点。第二，D3 对浏览器 DOM 事件的原生支持使得实现刷选联动、语义缩放等复杂交互成为可能，这对 Shneiderman [13] 倡导的"概览→缩放→细节"分析范式至关重要。第三，D3 生成标准 SVG/HTML 元素，可直接嵌入任何 Web 应用。当然，D3 的代码驱动特性也给质量检测提出了新要求：代码的视觉效果需要实际运行后才能确定。这正是 Design Lint 引入运行时渲染检测层的原因。

## 第二章 相关工作

### 2.1 可视化设计知识的组织与检索增强

#### 2.1.1 LLM 可视化生成中的知识获取

利用 LLM 生成可视化代码的研究按知识获取方式可分为三类。第一类完全依赖模型的参数化知识。Chat2Vis [5] 和 ChartGPT [6] 通过设计提示词模板引导 LLM 产出图表代码。NL4DV [14] 和 ncNet [15] 分别采用分析推断与序列到序列架构将自然语言映射到 Vega-Lite 规范。这些系统的设计质量受限于模型训练数据中可视化设计知识的覆盖范围。

第二类引入外部知识源的检索增强。LightVA [2] 首次在可视分析系统中引入 RAG，构建可视化设计知识库并在用户发起分析时检索相关知识注入提示词。LIDA [1] 通过多阶段推理引入中间推理步骤。InsightLens [16] 将检索增强思路从设计知识扩展到数据洞察领域。

第三类是新兴的多代理系统。ProactiveVA [3] 赋予系统主动分析能力。SmartMLVs [4] 实现多视图的自动生成与跨视图协调。

**尚未解决的问题。** 上述工作在知识检索方面存在共性局限：均采用逐条文本检索，将设计知识视为相互独立的条目。然而，可视化设计知识各维度之间存在丰富的关联——某种图表类型的选择往往需要同时考虑编码通道、颜色方案、交互设计乃至数据密度处理，这些因素之间的关联关系超出了文本相似度匹配的表达能力。此外，检索到的知识通常以散文式"建议"呈现，缺乏明确的适用条件和形式化规则。

#### 2.1.2 可视化设计知识的结构化

将设计经验转化为结构化、可计算的形式是可视化领域的重要研究方向。Mackinlay [17] 的 APT 系统首次体系化地提出编码选择准则——表达性（视觉通道应与数据类型的度量层级相匹配）和有效性（优先使用感知精度更高的通道）。Cleveland 和 McGill [7] 通过控制实验建立了各类视觉通道的感知精度排名。Heer 和 Bostock [18] 以大规模在线实验验证了这一排名的可靠性。Ware [19] 系统总结了人类视觉感知对可视化设计的约束。

在计算化方面，Draco [20] 将设计规则编码为逻辑约束，区分硬约束与软约束，通过约束求解器在 Vega-Lite 的规范空间中搜索满足所有约束的方案。Draco 2 [21] 进一步支持规则库的社区扩展和权重学习。CompassQL [22] 通过偏好打分实现可视化推荐，Dziban [23] 引入用户意图锚定。KG4Vis [24] 以知识图谱编码数据特征与图表类型之间的映射关系。

**VisArt 的定位。** 第一，在知识组织方面，我们采用图结构编码各类设计知识之间的关联，检索时沿图的边发现间接相关的技术与规则。第二，在应用方式上，与 Draco 通过约束求解直接生成 Vega-Lite 规范不同，我们将知识作为结构化上下文注入 LLM 提示词——利用大模型的代码生成能力处理 D3.js 的复杂代码合成，同时通过知识注入为其提供设计引导。第三，每条注入的知识都包含适用条件、规则内容和文献来源。

### 2.2 可视化质量评估与自动检测

#### 2.2.1 评估可视化设计质量的学术基础

可视化研究已为评估图表设计质量积累了大量有实验基础的依据。在编码有效性方面，Cleveland 和 McGill [7] 量化了不同视觉通道的判读精度差异。Kim 和 Heer [25] 以贝叶斯模型进行了更精细的建模。

在颜色使用方面，Borland 和 Taylor [8] 论证了彩虹色图亮度变化不均匀会在连续数据中制造视觉假象。Liu 和 Heer [26] 进一步证实感知均匀色板的数值读取优势。颜色的可区分性存在明确上限——Healey [27] 发现色相超过 7 种后搜索效率下降；Ware [19] 将上限确定为约 12 种；Haroz 和 Whitney [28] 从认知角度解释了这一阈值。

在无障碍方面，WCAG 2.1 [9] 规定图形对象与背景的颜色对比度不低于 3:1。Marriott 等 [29] 在可视化语境下讨论了此标准的重要性。Okabe 和 Ito [30] 提出了面向色觉障碍用户的配色原则。在视觉杂乱度方面，Ellis 和 Dix [31] 的研究表明重叠程度超过约 30% 时分析准确率显著下降。Kindlmann 和 Scheidegger [32] 从代数角度形式化了可视化设计误差。

这些理论成果为判断图表质量提供了坚实依据，但长期依赖设计者的个人经验加以应用。

#### 2.2.2 可视化 Linting

McNutt 和 Kindlmann [33] 首次提出"可视化 Linting"的概念——用程序自动检查图表的常见设计问题，类似于代码编辑器标出潜在错误。他们在 Matplotlib 上实现了概念验证。

VizLinter [10] 是目前功能最完整的可视化检测工具，面向 Vega-Lite 定义了 13 条检查规则，覆盖编码合法性、标记类型适配和通道使用合理性。VizLinter 可直接解析 Vega-Lite 的 JSON 结构来判断编码映射是否合理，并在检测之外实现了自动修复。GeoLinter [11] 将 linting 专门化于等值线地图。Ceneda 等 [34] 综述了可视分析过程中的自动化引导机制。

**尚未解决的问题。** 现有 linting 工具均面向声明式规范设计。在声明式系统中，图表的设计决策被显式记录在规范文件中，检测工具只需解析这些声明即可判断对错。然而，当 LLM 生成 D3.js 代码时，设计决策隐含在比例尺定义、数据绑定和 DOM 操作等代码逻辑中。一段 D3 代码的最终视觉效果——包括颜色的实际对比度、标记的重叠程度、色相的实际种类数——只有在代码运行并渲染出 SVG 后才能确定。目前不存在面向代码驱动可视化的自动化质量检测系统。

## 第三章 系统设计

基于上述分析，我们将 VisArt 的设计目标归纳为四点：

**G1：以结构化方式为 LLM 提供设计知识。** 将可视化设计准则组织为结构化知识库，在 LLM 生成图表时检索最相关的知识注入提示词，包含适用条件、规则内容和文献依据。

**G2：根据数据特征和分析任务智能匹配知识。** 自动分析数据特征和分析意图，筛选出真正相关的知识，避免遗漏关键规则，也避免不相关信息干扰模型判断。

**G3：自动检测生成图表的设计问题并量化评分。** 图表代码生成后自动进行多层面质量审查，输出量化的 DQS。发现严重问题时自动反馈给大模型修复。

**G4：使设计推理过程对用户透明。** 用户能看到系统检索了哪些知识、做了什么推理、发现了什么问题，从而理解设计决策的依据并在需要时人工干预。

### 3.1 可视化设计知识库

VisArt 的知识库包含 10 个结构化知识文件，覆盖可视化设计的主要方面。

**设计规则。** 包含 16 条硬规则和 26 条软规则。硬规则代表不可违反的设计底线，每条均源自经同行评审的学术研究。例如，"名义型数据不可使用亮度或大小通道编码"基于 Mackinlay [17] 的表达性原则；"连续数据不可使用彩虹色图"基于 Borland 和 Taylor [8] 的实验证据；"图形元素颜色对比度不低于 3:1"源自 WCAG 2.1 [9]。软规则编码设计偏好：如"数值数据优先用位置通道编码"（依据 Cleveland-McGill [7]）、"数据量超过一万条时建议聚合显示"（依据 Ellis 和 Dix [31]）。每条规则记录适用条件、规则内容、设计原理和出处文献。

**颜色知识。** 包含 11 组精选色板（标注色觉障碍安全性 [30]）、16 条颜色感知规则（如"分类配色色相不宜超过 12 种" [19]）以及 60 余条语义配色映射。语义配色基于 Lin 等 [37] 的研究——颜色与数据含义不一致时会产生认知冲突。

**交互与任务知识。** 包含四级数据密度策略、6 种多视图联动模式、语义缩放规范 [13]、四种数据类型的编码通道推荐排名 [7]，以及 Brehmer 和 Munzner [38] 任务分类的结构化表达。

上述知识以知识图谱形式组织，构建了**论文→分析任务→设计技术**的三层关联网络。不同论文中的任务若使用了相同的设计技术，图中便自然形成跨论文的关联——使检索不局限于与查询直接匹配的条目，还能沿关联边发现间接相关的设计信息。

### 3.2 检索与知识注入

当用户输入分析需求后，系统通过三阶段检索与组织设计知识。

**语义检索。** 计算用户查询和图谱中所有任务节点的语义向量，通过余弦相似度匹配找出最相关的若干任务。

**图遍历扩展。** 对检索到的每个任务节点，沿图的关联边进行两跳遍历：第一跳发现该任务关联的设计技术和来源论文；第二跳从设计技术出发，发现使用同一技术的其他任务。此阶段不仅返回与查询直接匹配的知识，还发现通过关联关系间接相关的知识——例如：查询"展示传感器数据中的异常模式"→ 匹配到"异常高亮"任务 → 发现其关联的"条件颜色编码"技术 → 再发现另一篇论文中使用该技术的"阈值监控"任务及其数据密度处理建议。

**约束推理与知识融合。** 图检索结果与颜色感知、交互设计、设计约束三条知识通道融合。融合前，约束推理引擎自动分析用户数据——识别各字段的数据类型（定量、序数、名义、时序）、唯一值数量、数据总量。同时参照 Brehmer 和 Munzner [38] 的任务分类框架推断分析意图。基于这些分析，引擎从 42 条规则中筛选出当前适用的子集（通常 8-15 条），连同数据特征、任务推断和编码推荐组装为结构化上下文，注入大模型提示词。

### 3.3 多角色生成架构

VisArt 将生成过程分为三个独立角色。

**评审者（Critic）。** 综合检索到的设计知识和约束推理引擎的上下文，评估可视化方案的合理性。评审输出引用具体规则和来源文献——例如"依据 Cleveland-McGill 的通道精度排名，建议将销售额映射到位置通道；当前方案的彩虹色图违反 HC-COLOR-002"。

**生成者（Generator）。** 基于评审意见和知识上下文生成 D3.js 可视化代码。

**修复者（Refiner）。** 接收用户修改意见和 Design Lint 检测到的质量问题，在保持核心设计的前提下定向修正问题。

三角色分离使用户可分别查看每个角色的推理过程和决策依据（G4）。

## 第四章 Design Lint：面向 D3.js 的自动化质量检测

### 4.1 设计动机

如第二章所述，现有可视化 linting 工具面向声明式规范设计，通过解析 JSON 配置中的编码声明来判断对错。然而 D3.js 代码不存在显式的编码声明——数据类型与视觉通道之间的映射关系散布在比例尺定义、数据绑定和 DOM 操作的各处逻辑中。要了解一段 D3 代码的最终视觉效果，只有在代码实际运行并渲染出 SVG 后才能确定。

Design Lint 正是为此设计，通过三层递进的检测架构应对代码驱动可视化的质量评估挑战。

### 4.2 三层检测架构

**第一层：静态代码分析。** 在代码运行前，通过正则匹配和抽象语法树分析识别无需运行即可确认的问题。当前实现包含以下检测项：

- **LINT-S02：比例尺域硬编码。** 检测代码中是否存在将比例尺域（domain）写为固定数值的情况。硬编码的域在数据更换时无法自适应，违反数据驱动原则 [32]。
- **LINT-S04：使用已知有缺陷的色图。** 检测是否使用了彩虹色系（如 Rainbow、Jet、HSL 色轮）作为连续色阶，依据 Borland 和 Taylor [8] 的实验证据。
- **LINT-S06：标记尺寸阈值。** 检测生成代码中设定的标记尺寸是否低于 2×2 像素。低于此阈值的标记不具备可辨识性 [19]。

**第二层：运行时渲染分析。** 在代码运行并渲染出 SVG 后，对实际渲染结果进行检测。这是应对代码驱动可视化质量检测的关键创新——以下信息只存在于运行后的渲染结果中：

- **LINT-R03：颜色对比度合规性。** 提取每个 SVG 图形元素的填充颜色和背景颜色，计算 WCAG 2.1 定义的亮度对比度，判断是否达到 3:1 的下限 [9]。
- **LINT-R06：色相可区分性。** 统计渲染结果中实际使用的不同色相数量，判断是否超过 Ware [19] 确定的 12 种上限。
- **LINT-R02：标记重叠程度。** 计算相邻 SVG 元素的边界框重叠面积比例，判断重叠程度是否超过 30% 的可读性阈值 [31]。
- **LINT-R05：图例和坐标轴完整性。** 检查渲染结果中是否存在图例和带有标签的坐标轴。

**第三层：设计规则合规性。** 对照知识库中的硬规则和软规则，检查整体设计方案在语义层面的合理性——例如编码通道与数据类型是否匹配、图表类型与分析任务是否适配。

### 4.3 设计质量分数（DQS）

三层检测的结果汇总为一个 0-100 的设计质量分数。计算方式如下：

- 基础分数 100 分
- 每项硬规则违反扣 10 分
- 每项软规则违反扣 3-5 分（依据规则权重）
- 运行时检测的量化指标（对比度合规率、重叠程度等）按比例影响分数

DQS 分为五个等级：A（≥85）、B（≥70）、C（≥55）、D（≥40）、F（<40）。当 DQS 低于预设阈值时，系统自动将检测结果以结构化形式反馈给修复代理，触发修复流程。

### 4.4 修复机制

修复代理接收结构化的违规报告（包含违规编号、层级、问题描述和修复建议），在保持图表核心设计的前提下进行定向修正。修复后重新运行检测，更新 DQS。系统最多执行两轮自动修复——超过两轮仍未达标的问题通常需要根本性的设计方案调整（如更换图表类型或数据聚合策略），这超出了代码微调的范畴。

## 第五章 案例分析

为验证 VisArt 两项核心贡献的实际效果，我们使用 LightVA [2] 论文中的两个评估数据集开展案例分析：Auto MPG 汽车性能数据集（406 条记录，9 个属性）和 Superstore 零售交易数据集（3,312 条记录，21 个字段）。选用相同数据集便于在相同条件下审视 VisArt 的贡献。

### 5.1 案例一：汽车性能多维比较

**分析任务。** 用户输入："My goal is to find a car that performs relatively well across all metrics." 该任务要求在多个性能指标之间综合权衡以筛选均衡表现的车型，属于 Brehmer-Munzner 分类中的多维比较任务。

**知识检索。** 约束推理引擎自动识别出五个定量字段（Miles per Gallon、Horsepower、Weight、Acceleration、Displacement）和一个含三个类别的名义型字段（Origin）。基于多维比较的任务特征，引擎激活了编码有效性规则（SC-EFF-001）、颜色感知规则（HC-COLOR-003）和交互规则。GraphRAG 检索匹配到与"多维性能比较"相关的任务，通过图遍历发现了平行坐标图（Parallel Coordinates）作为关联设计技术。

**生成结果。** 系统生成了一张包含五个定量维度的交互式平行坐标图，以三种颜色区分产地（USA、Europe、Japan），支持坐标轴区间刷选。

**质量检测（DQS = 72/C）。** Design Lint 检测到比例尺域硬编码（3 处，静态层）、部分标记尺寸不足（静态层）和约 60% 的折线交叉重叠（运行时层）。其中重叠问题是平行坐标图在中等数据规模下的固有挑战，仅在代码运行后方可量化。

**分析价值。** 此案例展示了 GraphRAG 通过图遍历发现平行坐标图这一设计方案的能力——单纯的文本检索不太可能将"多维性能比较"直接关联到"平行坐标图"，因为二者在文本层面的相似度不高；但在图谱中，"比较任务"节点与"平行坐标图"技术节点之间存在明确的关联边。此外，Design Lint 的运行时层检测到了 60% 的折线重叠——这是静态代码分析无法发现的问题，因为重叠程度取决于数据分布和渲染结果。

### 5.2 案例二：零售数据时序趋势分析

**分析任务。** 用户输入："Explore the temporal trends of product sales for each category." 该任务要求展示各产品品类的销售额随时间的变化趋势。在此前的一些工作中，完成相同层级的分析通常需要多步交互——先观察总体趋势，再按品类分拆 [2]。

**知识检索。** 约束推理引擎识别出 Order Date 为时序字段、Sales 为定量字段、Category 为含三个类别（Furniture、Office Supplies、Technology）的名义型字段。Scientific Audit 面板的推理记录显示：系统依据 GraphRAG 中"Trend Monitoring"相关论文的检索结果判断折线图为时序数据的最佳选择，同时参照 Cleveland-McGill 层级建议使用小多组（small multiples）布局使各品类趋势可独立比较。

**初始生成（DQS = 65/C）。** 系统生成了一组三行的小多组面积折线图。Design Lint 检测到 2 项错误和 4 项警告：比例尺域硬编码（7 处）、2 个标记对比度低于 WCAG 3:1 标准、8 个标记尺寸不足、以及缺少交互过滤和提示框。

**修复后（DQS = 75/B）。** 修复代理将硬编码域替换为数据驱动计算，调整颜色使对比度达标，加深背景色调满足 WCAG 要求。修复后 DQS 提升 10 分，硬编码和对比度问题消除。修复后的小多组布局清晰展示了三个品类的独立趋势。

**分析收获。** Key Insights 自动指出三个品类均呈现 Q4 季节性销售高峰（与假日促销有关），Technology 品类波动最大，Office Supplies 增长最平稳。

**此案例的意义。** 第一，展示了知识检索的正确性——查询包含"temporal trends"和"each category"两个分析意图，系统自动识别时序字段和分类字段，并将折线图和小多组布局作为推荐方案（与案例一中对分类数据拒绝使用折线图形成对比，说明系统根据数据类型做出不同推荐而非一刀切）。第二，展示了 Design Lint 的修复过程——从 DQS 65 到 75 的提升伴随着对比度和硬编码两类问题的消除，表明"生成→检测→修复"闭环的有效性。

### 5.3 案例三：零售数据销售-利润-折扣关联分析

**分析任务。** 用户输入："Analyze the correlation between sales and profit, considering the impact of discount." 该任务要求在分析销售额与利润之间关系的同时展示折扣因素的影响，涉及三个变量的联合可视化。在此前的一些工作中，完成同等深度的分析需要专家通过三轮独立交互逐步深入 [2]。

**知识检索。** 约束推理引擎识别出 Sales、Profit、Discount 三个定量字段，并检测到 Discount 有 14 个不同级别（0%至80%），这对颜色编码的可区分性构成挑战。知识面板显示系统激活了颜色视觉缺陷规则（HC-COLOR-CVD）、扩展色板亮度变化规则、对比度规则和视觉反馈规则。Scientific Audit 面板记录的推理判断气泡散点图为最佳方案：将 Sales 和 Profit 映射到 X/Y 轴位置（精度最高的通道 [7]），以颜色渐变编码 Discount，以气泡大小编码数据密度。

**初始生成（DQS = 62/C）。** 这是三个案例中检测到问题最多的一次——2 项错误和 6 项警告：比例尺域硬编码、922 个标记对比度低于 WCAG 3:1、14 种颜色超过推荐上限 12 种 [19]、约 26% 的标记重叠、缺少交互过滤。

**修复后（DQS = 72/C）。** 修复代理修复了对比度问题并调整颜色渐变方案。DQS 提升 10 分。值得注意的是，14 种颜色超限和 26% 重叠问题在修复后仍然存在——前者是折扣值本身有 14 个离散级别所致，后者是 3,312 条数据在二维散点图中的固有挑战，需要通过数据聚合或密度图等根本性方案调整来解决，超出了代码微调的范畴。

**分析收获。** Key Insights 识别出一个关键模式——当折扣率超过 20% 时，大量商品的利润跌入负值区域。高折扣商品集中分布在零利润线以下。同时 Technology 品类在低折扣区间利润最高，Furniture 对折扣敏感度最大。

**此案例的意义。** 第一，展示了约束推理引擎对多变量查询的理解能力——用户在一个查询中同时包含三个变量，引擎自动识别并分配到不同的视觉通道，而非要求用户逐步分解。第二，展示了 Design Lint 发现了仅凭肉眼或代码审查难以量化的问题：922 个标记的对比度不达标和 14 种颜色超出可区分上限。这类问题在不具备自动检测机制的系统中不会被发现。第三，诚实地展示了 Design Lint 的局限——某些问题（颜色数量由数据决定、高密度重叠）不是代码微调可以解决的，表明自动修复有其边界。

### 5.4 案例比较与讨论

三个案例的 DQS 变化汇总如下：

| 案例 | 初始 DQS | 修复后 DQS | 提升 | 消除的问题 | 保留的问题 |
|---|---|---|---|---|---|
| 汽车多维比较 | 72 (C) | —（未触发修复） | — | — | 60% 重叠 |
| 时序趋势 | 65 (C) | **75 (B)** | +10 | 硬编码 × 7，对比度 × 2 | 标记尺寸 |
| 销售-利润-折扣 | 62 (C) | **72 (C)** | +10 | 硬编码，对比度 × 922 | 14 色超限，26% 重叠 |

从以上结果可以观察到以下规律：

**"生成→检测→修复"闭环对代码层面的问题修复最为有效。** 硬编码和对比度不达标属于代码层面的缺陷，修复代理可以直接定位和修正。而重叠和颜色种类超限涉及图表设计方案的根本性选择，不是替换几行代码可以解决的。这表明 Design Lint 的价值不仅在于提升 DQS，还在于明确地指出**哪些问题可以自动修复、哪些需要人工决策**。

**知识检索为图表类型选择提供了可追溯的依据。** 三个案例中，系统分别选择了平行坐标图、小多组折线图和气泡散点图，每种选择都有 GraphRAG 的检索记录和 Scientific Audit 的推理支撑。用户可以查看这些依据并在不同意时进行人工调整。

**运行时检测层发现了其他方式无法发现的问题。** 922 个标记的对比度不达标、60% 的折线重叠、26% 的散点重叠——这些问题只有在代码运行并渲染出 SVG 后才可量化。静态代码分析和声明式规范检查均无法覆盖。

同时，现有的工作在几方面存在差距，VisArt 在此做出了补充：一是在知识检索方面，现有系统采用逐条文本检索，VisArt 通过图结构组织知识并利用图遍历发现间接关联；二是在质量保障方面，现有系统不提供生成后的质量反馈，VisArt 的 Design Lint 实现了自动化的多层检测和量化评分；三是在交互能力方面，多数现有系统生成静态图表，VisArt 利用 D3.js 生成原生支持交互的可视化。

## 第六章 讨论

### 6.1 Design Lint 的边界

案例分析表明，Design Lint 对代码层面的问题（硬编码、对比度不足）修复效率较高，但对涉及设计方案选择的问题（颜色种类超限、高密度重叠）效果有限。这一局限的根源在于：代码层面的问题可以通过定位和替换特定代码段来修复，而设计方案层面的问题需要根本性的策略调整——例如将连续色阶离散化为分组色阶、将散点图替换为密度图或六边形分箱图。未来工作可以探索在检测到此类问题时自动推荐替代方案（如建议"使用分箱聚合策略"），而不仅限于在现有代码基础上微调。

### 6.2 知识库的覆盖范围

当前知识库包含 42 条设计规则、11 组色板和基于 212 篇论文构建的知识图谱。虽然涵盖了可视化设计的核心方面，但在地理可视化、网络可视化和三维可视化等专门领域的覆盖尚不充分。知识库采用模块化的 JSON 格式，支持通过添加新的知识文件进行扩展，无需修改系统架构。

### 6.3 DQS 评分体系的局限

DQS 评分基于已有学术文献中的设计准则，侧重于设计规范的合规性而非用户主观的审美偏好。一张 DQS 得分较高的图表在设计规范层面是合理的，但不一定是用户心目中"最好看"的图表。此外，当前的扣分权重（硬规则 10 分，软规则 3-5 分）基于文献中的严重程度分级，尚未通过大规模用户研究加以验证。

### 6.4 系统的适用范围

VisArt 当前面向表格型数据生成二维统计图表。对于需要专业领域知识的可视化任务（如医学影像可视化、金融时序分析），系统需要扩充对应领域的知识库。此外，Design Lint 当前检测 SVG 格式的渲染结果；若系统扩展到 Canvas 或 WebGL 渲染，检测机制需要相应调整。

## 第七章 结论

本文提出 VisArt，一个从设计知识检索增强和生成后质量评估两个方面提升 LLM 可视化生成质量的系统。在知识检索方面，我们构建了以知识图谱组织的可视化设计知识库，通过图遍历和约束推理引擎为 LLM 提供结构化的设计引导。在质量评估方面，我们提出了面向 D3.js 代码的三层 Design Lint 系统，实现了代码层、运行时渲染层和设计规则层的自动化检测，并输出量化的设计质量分数（DQS）。

基于 Auto MPG 和 Superstore 两个数据集的案例分析表明：GraphRAG 的图遍历能够发现与查询间接相关的设计技术和规则；Design Lint 的运行时检测层能够发现仅靠静态分析无法量化的渲染问题（如对比度不达标和标记重叠）；"生成→检测→修复"闭环对代码层面的问题修复有效（DQS 平均提升约 10 分）但对设计方案层面的问题效果有限。这些结果验证了两项贡献的实际价值，同时诚实地呈现了系统的边界。

未来工作将从三个方面展开：一是在 Design Lint 检测到设计方案层面的问题时自动推荐替代方案；二是扩充知识库对地理和网络可视化等专门领域的覆盖；三是通过大规模用户研究验证和优化 DQS 的权重体系。
