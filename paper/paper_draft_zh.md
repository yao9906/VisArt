# VisArt：基于图结构知识检索与运行时渲染审查的 LLM 可视化生成质量保障系统

---

## 摘要

大型语言模型（LLM）能够将自然语言描述转换为可执行的可视化代码，但已有研究表明，即使代码能够成功运行，生成图表的设计质量仍然堪忧——Ford 和 Rios [38] 发现在 Text2Chart31 基准上仅有 7.2%–33.3% 的 LLM 生成图表满足基本的色觉障碍无障碍准则。这一现状揭示了两个尚未解决的方法论挑战：首先，可视化设计知识高度耦合（数据类型→视觉通道→感知规则相互制约），现有基于扁平向量检索的方法无法捕获这种跨维度的结构化依赖；其次，现有的可视化 Linting 工具（如 VizLinter [10]）依赖对 Vega-Lite 等声明式规范的静态解析，无法应用于 LLM 广泛生成的命令式 D3.js 代码——其视觉映射隐藏在运行时逻辑中，渲染前不可确定。

为此，本文提出 **VisArt**，一个从"知识注入"到"运行时评估"全链路保障 LLM 图表生成质量的系统。VisArt 的流水线包含三个核心步骤：**Step 1（评审与知识检索）**，基于知识图谱的多跳结构检索引擎从 212 篇论文构建的设计知识图谱中提取跨维度关联知识，约束推理引擎根据数据特征筛选适用规则并生成结构化上下文注入 LLM 提示词；**Step 2（代码生成）**，LLM 基于结构化上下文生成 D3.js 可视化代码；**Step 3（Design Lint 审查与修复）**，首个面向命令式可视化代码的三层自动审查框架——通过在无头渲染环境中执行代码并对 SVG DOM 树进行结构探查，量化评估仅在运行时才能确定的感知指标（如标记重叠率、WCAG 颜色对比度），输出设计质量分数（DQS），并在评分低于阈值时自动驱动修复。基于两个公开数据集的案例分析表明，VisArt 的闭环机制能有效纠正 LLM 的隐性设计违规。

---

## 第一章 引言

自然语言驱动的图表生成正在改变数据可视化的工作流程。LIDA [1]、Chat2Vis [5] 等系统允许用户通过自然语言直接生成图表代码。LightVA [2] 和 ProactiveVA [3] 引入多智能体架构，将 LLM 集成到可视化分析的规划与交互生命周期中。Ford 和 Rios [38] 提出轻量级多智能体流水线（Drafting→Execution→Repair），在 Text2Chart31 基准上将执行错误率从约 15% 降至 4.5%，证明多步修复代理在提升代码可执行性上的有效性。

然而，**代码能运行不等于图表质量好**。Ford 和 Rios [38] 的进一步分析发现，在成功运行的图表中仅 7.2%–33.3% 满足基本色觉障碍准则，6% 包含数据幻觉。LLM 极易生成"语法正确但视觉感知存在严重缺陷"的图表——对连续数据使用彩虹色图 [8]，在有限空间内绘制数千散点导致严重重叠 [31]，或图形元素色彩对比度低于无障碍标准 [9]。现有系统在保障设计质量方面存在两个方法论挑战：

**挑战一：设计知识检索缺乏结构化约束关联。** 高质量可视化设计是复杂的约束求解过程：选定图表类型会触发颜色可区分性 [19]、标记重叠处理 [31]、交互规范 [13] 的连锁约束。现有基于余弦相似度的扁平检索仅能匹配文本语义近似的条目，无法捕获"任务→图表→感知规则"的逻辑依赖。Ford 和 Rios [38] 的流水线中不包含结构化设计知识检索机制，其 Drafting Agent 仅使用通用提示词。

**挑战二：缺乏面向命令式代码的运行时质量反馈。** VizLinter [10] 面向 Vega-Lite JSON 的静态解析。但 D3.js 的数据映射逻辑分散在 JavaScript 控制流中，图表的真实视觉表现——标记间距、颜色对比度、文字可读性——在代码执行并渲染出 SVG DOM 树之前处于不可确定状态。Ford 和 Rios [38] 的 Re-Writer Agent 只处理 Python 解释器的运行时异常（runtime exception），不涉及渲染后的视觉质量评估。

为解决上述挑战，本文提出 VisArt，提供以下三项贡献：

- **贡献一**：基于知识图谱的多跳结构检索约束引擎——通过图谱多跳遍历实现语义检索与结构检索的协同，发现分散在不同维度中的跨维度设计关联。
- **贡献二**：首个面向 D3.js 命令式代码的三层运行时审查框架（Design Lint）——通过无头渲染 + DOM 探查量化高阶感知指标，输出设计质量分数（DQS）。
- **贡献三**：评估驱动的闭环修复机制——基于 Design Lint 结构化报告自主定向修复违规代码。

## 第二章 相关工作

### 2.1 LLM 驱动的可视化生成

#### 2.1.1 基于参数化知识的生成

Chat2Vis [5] 和 ChartGPT [6] 依赖模型预训练参数化知识生成图表。NL4DV [14] 和 ncNet [15] 通过分析推断或神经翻译映射到 Vega-Lite。这类系统的设计质量上限受制于训练数据覆盖程度，无法确保输出符合经实验验证的感知原则。

#### 2.1.2 基于检索增强与多智能体的生成

LIDA [1] 引入多阶段推理链。LightVA [2] 提出多智能体框架将可视分析分解为规划、执行和评估的协同。SmartMLVs [4] 实现多视图协调生成。Ford 和 Rios [38] 提出 Drafting + Re-Writer 的轻量级多智能体流水线，通过执行错误反馈驱动迭代修复（最多三轮），在 Text2Chart31 上将执行错误率降至 4.5%。然而他们也指出，执行成功并不等同于设计质量，仅 7.2%–33.3% 的图表满足色觉障碍准则。

**本文方法的定位。** 上述工作在检索设计知识时将各维度知识作为独立文本条目检索，难以捕获跨维度的结构关联。VisArt 通过知识图谱的多跳遍历解决此问题。与 Ford 和 Rios [38] 仅关注代码执行成功率不同，VisArt 进一步在渲染后对视觉质量进行定量评估。

### 2.2 可视化设计知识的结构化表达

Mackinlay [17] 提出编码的表达性与有效性准则。Cleveland 和 McGill [7] 建立视觉通道感知精度排名。Draco [20] 将设计规则编码为 ASP 约束。KG4Vis [24] 以知识图谱编码数据特征与图表类型的映射。本文在 KG4Vis 基础上将编码规则、颜色感知、交互规范和任务分类等多维度知识融合进同一图谱，使多跳遍历能发现跨维度关联。

### 2.3 可视化质量评估与 Linting

颜色可区分性存在认知上限 [19, 27, 28]。Borland 和 Taylor [8] 证明彩虹色图产生感知假象。WCAG 2.1 [9] 规定非文本图形对比度不低于 3:1。Ellis 和 Dix [31] 表明重叠超 30% 分析准确率显著下降。McNutt 和 Kindlmann [33] 提出"可视化 Linting"概念。VizLinter [10] 面向 Vega-Lite 实现 13 条规则。GeoLinter [11] 专门化于地图。**现有 Linting 工具均面向声明式规范的静态解析，不支持运行时分析。** Ford 和 Rios [38] 使用 LLM-as-a-judge 评估色觉障碍合规性，但其方法依赖多模态 LLM 对图像的主观判断，而非对 DOM 的精确定量计算。

## 第三章 系统概述

根据相关工作中存在的研究缺口，并参考 Bach 等人 [36] 的可视化任务分类，我们为 VisArt 确定了以下四个设计目标：

**G1：以结构化方式为 LLM 提供多维度关联的设计知识。** 系统应不仅匹配语义近似的知识条目，还通过知识图谱的关联边发现跨维度的间接相关知识，为 LLM 提供具有逻辑支撑的结构化上下文。

**G2：根据数据特征和分析意图智能筛选适用约束。** 系统应自动分析数据的字段类型、唯一值数量和数据总量，从设计规则库中筛选当前适用的子集，注入精准上下文。

**G3：在图表代码渲染后自动执行定量质量审查并驱动修复。** 系统应在代码渲染后对 SVG DOM 进行定量检测，输出量化分数，低于阈值时自动驱动修复。

**G4：使设计推理过程对用户透明可查。** 从知识检索到评审意见到代码生成，每一步推理应对用户可见。

图 1 展示了 VisArt 的整体流水线。系统由三个步骤组成：**Step 1** 评审与知识检索（G1, G2, G4），**Step 2** 代码生成（G4），**Step 3** Design Lint 审查与修复（G3）。若 Step 3 的 DQS 低于阈值，系统自动将违规报告反馈到 Step 2 进行修复，最多迭代两轮。

## 第四章 方法

图 1 展示了 VisArt 流水线的高层概览。流水线包含三个核心步骤。首先，**评审代理（Critic Agent）** 接收用户的自然语言需求和数据集，调用知识图谱检索引擎提取结构化设计知识，输出评审意见。其次，生成的评审意见和知识上下文被传递给 **生成代理（Generator Agent）**，生成 D3.js 可视化代码。最后，生成的代码在无头渲染环境中执行，**Design Lint** 对渲染后的 SVG DOM 进行三层定量检测；若检测发现问题（DQS < 70），违规报告被传回生成代理进行定向修复。修复后重新执行检测，最多迭代两轮。下面详细描述每个步骤。

### 4.1 Step 1：评审代理与知识图谱检索

#### 4.1.1 知识库构建

VisArt 的知识库由 10 个模块化 JSON 文件组成，覆盖五个知识维度：

- **编码规则维度：** 16 条硬规则（不可违反的设计底线，每条有实验文献支撑）和 26 条软规则（带权重的设计偏好）。例如，"分类数据不可以亮度或大小通道编码"基于 Mackinlay [17]；"连续数据不可使用彩虹色图"基于 Borland 和 Taylor [8]；"图形元素颜色对比度不低于 3:1"源自 WCAG 2.1 [9]。
- **颜色感知维度：** 11 组标注色觉障碍安全性的色板 [30]、16 条颜色感知规则、60 余条语义配色映射 [37]。
- **交互规范维度：** 四级数据密度交互策略、6 种多视图联动模式、语义缩放规范 [13]。
- **任务−技术映射维度：** 基于 212 篇可视化论文构建的"论文→分析任务→设计技术"三层关联网络。不同论文中的分析任务若关联同一设计技术，图谱中自然形成跨论文的结构关联边——这是实现结构检索的基础 [24]。
- **数据特征感知维度：** 各字段数据类型对应的编码通道推荐优先级和数据量级与渲染策略的映射关系 [31]。

#### 4.1.2 语义检索与结构检索的协同

当用户输入分析需求后，系统通过三个阶段检索设计知识。

**阶段一：语义检索（Semantic Retrieval）。** 使用 Gemini Embedding API 将用户查询编码为 768 维稠密向量，与知识图谱中所有分析任务节点的预计算嵌入进行余弦相似度匹配，返回 top-k（默认 k=3）语义最近的任务节点作为种子节点。此阶段对应传统 Naive RAG。

**阶段二：结构检索——多跳图遍历（Structural Retrieval）。** 对每个种子任务节点，沿知识图谱的类型化关联边进行两跳遍历。第一跳（任务→技术）沿 `USES_TECHNIQUE` 边发现该任务关联的设计技术节点及其来源论文（`DEFINES_TASK` 边）；第二跳（技术→任务）沿反向 `USED_BY` 边找到所有使用同一技术的其他任务节点（最多 3 个）及其设计经验。例如，查询"多维性能比较"→语义匹配到"多属性均衡比较"任务→第一跳发现"平行坐标图"技术→第二跳从该节点反向发现"高维折线重叠处理"和"坐标轴刷选交互规范"——这些知识在文本语义层面与查询不接近，但在图谱设计逻辑中紧密相连。

**阶段三：约束推理与知识融合（Constraint Reasoning）。** 图遍历结果与颜色感知、交互规范、编码约束三个知识维度融合。约束推理引擎自动分析用户数据——识别各字段的数据类型（数值型、序数型、分类型、时序型）、唯一值数量和数据总量——并参照 Brehmer-Munzner 任务分类 [36] 推断分析意图。基于这些分析，引擎从 42 条设计规则中筛选当前适用的子集（通常 8-15 条），组装为结构化上下文。

#### 4.1.3 评审输出

评审代理综合图谱检索结果和约束推理上下文，输出一份结构化的设计评审意见，包括推荐图表类型、编码通道分配、颜色方案和交互设计。评审意见以引用形式呈现推理依据——例如"依据 Cleveland-McGill [7] 通道精度排名第一位选择位置通道编码 Sales；候选方案的彩虹色图引用违反 HC-COLOR-007"。评审输出记录在界面右侧的 Scientific Audit 面板中（**G4**）。

### 4.2 Step 2：生成代理

生成代理接收评审意见和结构化知识上下文，调用 LLM 生成 D3.js 可视化代码。具体而言，系统提示词包含以下组件：

> **系统提示词（摘要）：**
> 你是一位 D3.js 可视化专家。基于以下结构化设计上下文生成完整的 D3.js 代码。代码必须：(1) 使用全局变量 `d3` 而非 import；(2) 从提供的数据动态计算比例尺域；(3) 遵循设计上下文中列出的所有编码和颜色约束。

> **用户指令：**
> 分析需求：{用户输入}
> 数据集概要：{字段名、类型、前 5 行}
> 设计上下文：{约束推理引擎输出的结构化规则集}
> 评审意见：{Critic 输出}

生成的代码在前端 D3 渲染器中挂载执行，生成 SVG 图形元素。渲染完成后，系统自动触发 Step 3 的 Design Lint 检测。

当 Design Lint 输出的 DQS < 70 时，系统自动生成修复提示词（包含违规编号、层级、问题描述和建议修复方式），注入生成代理进行定向修复。修复提示词格式如下：

> **修复指令（示例）：**
> 以下是 Design Lint 检测到的违规项，请在不破坏整体设计的前提下修复：
> - LINT-S01 (Error): 3 处比例尺域硬编码，建议替换为 d3.extent()
> - LINT-R03 (Warning): 2 个标记颜色对比度低于 WCAG 3:1，建议加深填充色

修复后重新执行三层检测。系统最多自动迭代两轮——超过两轮仍未达标的问题通常涉及方案的根本性调整（如从散点图切换到六边形分箱），超出代码微调范畴。此外，用户也可在界面文本框中输入自然语言修改意见，触发交互式修复。

### 4.3 Step 3：Design Lint——面向命令式代码的运行时审查

#### 4.3.1 问题界定

现有可视化 Linting 工具的前提是被检测对象以声明式规范存在——VizLinter [10] 直接读取 Vega-Lite JSON 中 `mark`、`encoding`、`scale` 字段判断规则匹配度。然而 D3.js 的映射关系分散在 `d3.scaleLinear().domain(...)` 的比例尺定义、`.data(data).join('circle')` 的数据绑定和 `.attr('fill', d => colorScale(d.field))` 的运行时计算中。颜色的最终渲染值、标记的几何边界框、SVG 元素间的空间重叠——这些都是 JavaScript 引擎完整执行代码后才能确定的运行时状态。Design Lint 通过三层递进架构应对这一挑战。

#### 4.3.2 第一层：静态代码分析（6 项规则）

代码执行前，通过正则表达式匹配和代码模式扫描识别确定性的代码级问题：

| 编号 | 检测内容 | 严重程度 | 依据 |
|------|----------|----------|------|
| **LINT-S01** | 比例尺域硬编码 `.domain([0, 100])` | Error | 数据驱动原则 [32] |
| **LINT-S02** | 缺少空数据防护 | Warning | 鲁棒性 |
| **LINT-S03** | 存在 `import`/`require` 语句 | Error | VisArt 沙盒约束 |
| **LINT-S04** | 标记半径硬编码 < 2px | Warning | 可辨识性 [19] |
| **LINT-S05** | 使用彩虹/Jet 色图 | Error | Borland & Taylor [8] |
| **LINT-S06** | 缺少悬停交互（tooltip） | Warning | 按需细节 [13] |

#### 4.3.3 第二层：运行时渲染分析——DOM 探查（9 项规则）

**这是 Design Lint 的核心创新层，也是现有 Linting 工具的盲区。** 系统将 D3.js 代码在无头渲染环境中执行，代码对 DOM 的所有操作完整运行后，生成 SVG 树挂载到文档中。随后对 SVG DOM 进行深度探查（DOM Inspection），提取每个图形元素的运行时状态：

| 编号 | 检测内容 | 方法 | 依据 |
|------|----------|------|------|
| **LINT-R01** | SVG 容器是否存在 | DOM 查询 | 渲染管线完整性 |
| **LINT-R02** | 是否存在图形标记 | DOM 查询 `circle/rect/path` | 数据绑定正确性 |
| **LINT-R03** | 颜色对比度 ≥ 3:1 | `getComputedStyle()` + WCAG 亮度公式 | WCAG 2.1 [9] |
| **LINT-R04** | 渲染后标记尺寸 ≥ 2×2px | `getBBox()` | 可辨识性 [19] |
| **LINT-R05** | 色相数量 ≤ 12 种 | `getComputedStyle()` 统计唯一色相 | Ware [19] |
| **LINT-R06** | 文字字体 ≥ 8px | `getComputedStyle()` 字号 | 可读性 |
| **LINT-R07** | 多色图表是否含图例 | DOM 查询图例元素 | 语义透明性 |
| **LINT-R08** | 是否含坐标轴 | DOM 查询 `.tick`/`.domain` 类 | 数值可参照性 |
| **LINT-R09** | 标记重叠率 ≤ 30% | `getBBox()` + 网格投影 | Ellis & Dix [31] |

**关键点：** LINT-R03 的颜色对比度评估在代码执行前不可能完成——颜色值可能通过 `d3.scaleOrdinal(d3.schemeTableau10)` 的序数色板或数据驱动函数动态计算，只有在代码执行后才能从 `getComputedStyle()` 的计算样式中读取真实值。LINT-R09 的重叠率完全取决于数据的空间分布和比例尺的 Range 设置，同一段代码在不同数据集与容器尺寸下的重叠率可能截然不同。

#### 4.3.4 第三层：设计规则合规性（后端约束层）

本层通过后端约束推理引擎将图表的整体设计方案与知识库中的 16 条硬规则和 26 条软规则比对，从设计语义层面评估合理性——例如检查编码通道与数据类型是否匹配 [17]、颜色方案是否色觉障碍友好 [30]。与前两层不同，本层检测对象不是代码结构或 SVG 几何属性，而是图表的整体设计语义。

#### 4.3.5 设计质量分数（DQS）

三层检测结果汇总为 0-100 的设计质量分数：以 100 分为基准，每项严重违规（error）扣 10 分，每项一般违规（warning）扣 3 分。DQS 划分为五个等级：A（≥90）、B（≥75）、C（≥60）、D（≥40）、F（<40）。

## 第五章 可视化界面

VisArt 的可视化界面由三列布局组成（图 X），支持从知识审查到图表生成到质量评估的全流程交互（**G4**）。

**知识面板（左侧）。** 显示当前检索到的设计知识，分为三个知识泳道：设计原则（Design Principles）、颜色与感知（Color & Perception）和交互规则（Interaction Rules）。每条命中规则显示主题、相似度分数、适用条件和来源文献，可展开查看完整内容。面板顶部的 Graph Trace 小窗以力导向图形式显示 GraphRAG 在知识图谱中的遍历路径。面板还提供数据浏览子选项卡，显示数据集行数、字段数和预览。

## 第六章 案例分析

为验证 VisArt 两项核心贡献的实际效果，我们使用两个公开数据集开展案例分析：Auto MPG 汽车数据集（406 条记录，9 属性）和 Superstore 零售数据集（3,312 条记录，21 字段）。两个数据集同时被 LightVA [2] 用于专家评估，便于在相同条件下展示系统效果。

### 6.1 汽车性能数据集：多维比较分析

**分析任务。** 用户输入："My goal is to find a car that performs relatively well across all metrics."

**Step 1 执行过程。** 约束推理引擎识别出五个数值型字段（MPG、Horsepower、Weight、Acceleration、Displacement）和一个含三个水平的分类字段（Origin）。语义检索匹配到"多属性均衡比较"任务节点；结构检索沿图谱关联边进行两跳遍历：第一跳发现平行坐标图作为关联技术；第二跳从该节点反向发现"刷选联动"和"颜色分组不超过五类"的规范——这些跨维度知识通过结构遍历被纳入上下文（**G1**）。

**Step 2 生成结果。** 系统生成包含五个维度的交互式平行坐标图，以三种颜色区分产地，支持坐标轴区间刷选。

**Step 3 Design Lint 检测（DQS = 72/C）。** 检测到：LINT-S01（硬编码 ×3，静态层）、LINT-R04（8 个标记渲染后过小，运行时层）、LINT-R09（约 60% 折线重叠，运行时层）。**重叠率 60% 是通过 `getBBox()` 对 406 条折线坐标点进行网格量化投影后获得的——同一段代码在 50 条数据下不存在重叠，在 406 条数据下才暴露（G3）。**

### 6.2 零售交易数据集

#### 6.2.1 品类销售时序趋势

**分析任务。** 用户输入："Explore the temporal trends of product sales for each category."

**知识注入与生成。** 约束推理引擎识别 Order Date 为时序字段、Category 为含三个水平的分类字段。结构检索第一跳发现折线图为关联技术，进而发现小多组布局作为推荐结构——系统在一次生成中直接产出分品类的三行小多组面积折线图。

**初始生成（DQS = 65/C）与修复（DQS = 75/B）。** 初始检测发现 LINT-S01（硬编码 ×7）、LINT-R03（对比度不达标 ×2）、LINT-R04（标记过小 ×8）、LINT-S06（缺少悬停提示）。修复代理将硬编码域替换为 `d3.extent()` 并调整颜色使对比度达标。修复后 LINT-S01 和 LINT-R03 消除，DQS 提升至 75（B 级）。

#### 6.2.2 销售-利润-折扣三维关联分析

**分析任务。** 用户输入："Analyze the correlation between sales and profit, considering the impact of discount."

**知识注入与生成。** 约束引擎检测到 Discount 有 14 个离散取值，激活"色相超过 12 种需改用连续色阶"规则。系统生成三维气泡散点图。

**初始生成（DQS = 62/C）与修复（DQS = 72/C）。** LINT-R03 标记 922 处对比度不达标（经 DOM 探查对 3,312 个元素逐一计算）、LINT-R05 报告 14 色超限、LINT-R09 检测 26% 重叠。修复后 LINT-S01 和 LINT-R03 消除。**LINT-R05（14 色超限）和 LINT-R09（26% 重叠）仍保留**——前者由 Discount 字段本身 14 个离散值决定，后者是 3,312 条数据的固有挑战，需调整聚合策略或颜色编码方案，超出代码微调范畴。

### 6.3 分析与讨论

| 测试任务 | 初始 DQS | 修复后 DQS | 提升 | 消除的问题 | 保留的问题 |
|---|---|---|---|---|---|
| 汽车多维比较 | 72 (C) | — | — | — | 60% 重叠 |
| 零售时序趋势 | 65 (C) | **75 (B)** | +10 | 硬编码 ×7，对比度 ×2 | 标记尺寸 |
| 零售三维关联 | 62 (C) | **72 (C)** | +10 | 硬编码，对比度 ×922 | 14 色超限，26% 重叠 |

**结构检索的价值。** 汽车案例中"多维比较"与"平行坐标图"文本语义不接近，但通过图谱任务→技术边成功关联（**G1**）。

**运行时检测的不可替代性。** 三组测试共通过 DOM 探查检测到：922 处对比度不达标、约 60% 折线重叠和约 26% 散点重叠——均经 `getComputedStyle()` 和 `getBBox()` 逐元素计算获得，静态分析和声明式规范检查均无法产出（**G3**）。

**闭环修复的边界。** 代码层问题（硬编码、对比度）修复成效显著（DQS 平均 +10）；方案层问题（高密度重叠、数据驱动的颜色超限）保留且 DQS 最高仅达 B 级。Design Lint 的价值不仅在于提升 DQS，更在于**明确区分"可自动修复的代码级问题"与"需要人工决策的方案级问题"**——这一区分能力是 Ford 和 Rios [38] 的 Re-Writer Agent（仅处理 Python 运行时异常）所不具备的。

## 第七章 讨论

**修复边界。** 两轮自动修复对代码级问题有效，对方案调整无能为力。未来可探索检测到方案问题时自动推荐替代方案（如"将散点图替换为六边形分箱热力图"）。

**与 Ford 和 Rios [38] 的互补性。** Ford 和 Rios [38] 关注代码执行成功率（从 15% 降至 4.5%），VisArt 关注渲染后的视觉质量。两者的闭环修复分别针对不同层次的问题：前者修复 Python 运行时异常，后者修复 D3.js/SVG 层面的视觉感知违规。未来可将两种机制集成为统一框架。

**知识库覆盖范围。** 当前以二维统计图表的通用设计准则为主，地理和网络可视化覆盖有限。模块化 JSON 结构支持按需扩展。

**DQS 权重体系。** 当前扣分权重基于文献严重程度分级，尚未经大规模用户研究验证。DQS 反映规范合规性，不直接等同于用户主观审美评价。

**泛化性。** 运行时 DOM 探查方案可推广到任何生成 SVG 的 JavaScript 可视化库（如 Plotly.js、ECharts）。面向 Canvas/WebGL 的扩展需引入像素级颜色采样替代 DOM 探查。图谱检索机制与目标代码形式无关。

## 第八章 结论

本文提出 VisArt，从设计知识检索增强和生成后质量评估两方面提升 LLM 可视化生成质量。知识图谱的多跳结构检索将语义匹配与结构关联结合，弥补扁平检索的覆盖盲区。Design Lint 的三层架构——静态代码层（6 项规则）、运行时 DOM 探查层（9 项规则，通过 `getBBox()` + `getComputedStyle()` 实现）和设计规则合规层——实现对命令式可视化代码的全面审查，填补了面向代码驱动可视化自动评估的方法论空白。

案例分析表明，结构检索发现语义检索无法覆盖的跨维度知识关联；运行时层通过 DOM 探查量化了渲染后的感知指标（922 处对比度不达标、60% 折线重叠、26% 散点重叠）；闭环修复对代码级问题有效（DQS 平均提升约 10 分），对方案级问题自动修复能力有限但明确区分了两类问题的边界。与 Ford 和 Rios [38] 的工作形成互补——他们证明多步修复可解决代码执行问题，VisArt 进一步证明渲染后的定量 DOM 审查可解决视觉质量问题。

未来工作将延伸到三个方向：(1) 方案级替代推荐——检测到方案问题时自动生成替代方案建议；(2) 扩充地理与网络可视化领域的专项知识；(3) 通过大规模用户研究对 DQS 权重体系进行实证标定。
